# N4: 审查规范 (全面审查)

## 1. 长驻 Node 进程适配检查
- [ ] **冷启动优化**：检查是否引入了过大的包（如大型库的整个导入）。
- [ ] **环境变量**：确保所有敏感信息都在 `.env` 中，且在 `packages/env` 中有类型定义。
- [ ] **Better-Auth 兼容性**：检查认证中间件是否会造成死循环或高延迟。
- [ ] **错误处理**：tRPC 必须抛出标准的 `TRPCError` 供前端捕获。

## 2. 安全性检查 (新增)
- [ ] **SQL 注入防护**：必须使用 Drizzle `sql` 模板参数化，禁止 `sql.raw` 拼接。
- [ ] **XSS 防护**：React 自动转义，但检查 `dangerouslySetInnerHTML` 使用。
- [ ] **CSRF 防护**：Better-Auth 内置，检查 `trustedOrigins` 配置。
- [ ] **敏感信息不硬编码**：检查代码中是否有 API Key、密码等。
- [ ] **权限验证完整**：protectedProcedure 正确使用，adminProcedure 权限检查。
- [ ] **文件上传安全**：文件类型验证、大小限制、病毒扫描（如需要）。
- [ ] **速率限制**：公开 API 是否有速率限制配置。
- [ ] **输入验证**：所有 tRPC input 是否使用 zod 验证。

## 3. 性能检查 (新增)
- [ ] **无 N+1 查询**：使用 `include` 或 `select`，避免循环中查询数据库。
- [ ] **无内存泄漏**：无全局变量存储大量数据，无未清理的定时器。
- [ ] **数据库查询优化**：大查询使用分页，添加必要索引。
- [ ] **前端代码分割**：大组件使用 `next/dynamic` 懒加载。
- [ ] **图片懒加载**：使用 `<img loading="lazy">`（本项目无 `next/image`）。
- [ ] **API 缓存策略**：查询使用 React Query 的 `staleTime`。
- [ ] **无不必要的重渲染**：使用 `React.memo`、`useMemo`、`useCallback` 优化。

## 4. 可访问性检查 (新增)
- [ ] **ARIA 标签完整**：按钮、输入框、模态框有正确的 ARIA 标签。
- [ ] **键盘导航支持**：所有交互元素可通过键盘访问。
- [ ] **颜色对比度达标**：文字与背景对比度 >= 4.5:1。
- [ ] **表单标签完整**：所有输入框有 `<label>` 关联。
- [ ] **焦点管理正确**：模态框打开时焦点正确管理。

## 5. 错误处理检查 (新增)
- [ ] **边界条件处理**：空数组、undefined、null 处理。
- [ ] **异常捕获完整**：try-catch 覆盖异步操作。
- [ ] **错误信息友好**：用户可见的错误信息清晰、有帮助。
- [ ] **日志记录完整**：关键操作记录日志，包含上下文信息。
- [ ] **用户提示清晰**：加载状态、成功/失败提示完整。

## 6. 代码质量检查 (新增)
- [ ] **代码注释充分**：复杂逻辑有注释说明。
- [ ] **命名规范统一**：变量、函数、组件命名清晰一致。
- [ ] **无重复代码**：提取公共逻辑到 hooks 或工具函数。
- [ ] **函数职责单一**：每个函数只做一件事。
- [ ] **TypeScript 类型完整**：无 `any` 类型，类型定义完整。
- [ ] **代码结构清晰**：文件和目录组织合理。

## 7. 测试检查 (新增)
- [ ] **单元测试覆盖**：核心逻辑有单元测试。
- [ ] **组件测试覆盖**：关键组件有组件测试。
- [ ] **测试用例充分**：覆盖正常路径和边界情况。
- [ ] **测试可维护**：测试代码清晰、易读。

## 审查流程

1. **自动化检查**
   - 运行 `pnpm check-types` 检查类型
   - 运行 `pnpm lint` 检查代码规范
   - 运行 `pnpm test` 检查测试

2. **手动审查**
   - 按照上述清单逐项检查
   - 记录发现的问题

3. **问题分级**
   - 🔴 **严重**：安全漏洞、数据丢失风险、必须修复
   - 🟡 **警告**：性能问题、代码质量问题、建议修复
   - 🟢 **建议**：优化建议、可选改进

4. **处理决策**
   - 有 🔴 严重问题：回退 N3 重修
   - 有 🟡 警告问题：评估后决定是否修复
   - 只有 🟢 建议：通过审查，记录建议

✓ 如果审查失败，回退到 N3 重修。
✓ 审查通过后执行 `git commit`。

  - BLOCK → commit 被拦截，根据 codex 反馈修复后重试
