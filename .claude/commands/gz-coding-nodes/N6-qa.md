### 2. 评分维度

#### 2.1 功能完整性 (30%)

**评分标准**:
- 100%: 所有需求点完美实现，边界条件处理完善
- 90%: 所有需求点实现，少量边界条件未处理
- 80%: 主要需求点实现，部分次要功能缺失
- 70%: 核心功能实现，但有明显功能缺陷
- 60%: 核心功能基本实现，但有 Bug
- < 60%: 功能不完整，无法使用

**检查项**:
- [ ] 所有用户故事已实现
- [ ] 边界条件处理（空值、极大值、极小值）
- [ ] 错误场景处理（网络错误、API 错误）
- [ ] 用户体验流畅（加载状态、成功/失败提示）

#### 2.2 代码质量 (25%)

**评分标准**:
- 100%: 代码规范完美，类型安全完整，注释充分
- 90%: 代码规范良好，类型安全完整
- 80%: 代码规范良好，少量类型使用 any
- 70%: 代码规范一般，有一些重复代码
- 60%: 代码规范较差，有明显问题
- < 60%: 代码质量差，难以维护

**检查项**:
- [ ] 代码规范统一（ESLint 无警告）
- [ ] TypeScript 类型完整（无 any）
- [ ] 注释充分（复杂逻辑有说明）
- [ ] 无重复代码（提取公共逻辑）
- [ ] 函数职责单一
- [ ] 代码结构清晰

#### 2.3 性能表现 (20%)

**评分标准**:
- 100%: 所有性能指标优秀（Lighthouse > 95）
- 90%: 所有性能指标良好（Lighthouse > 90）
- 80%: 主要性能指标达标（Lighthouse > 80）
- 70%: 性能基本达标，但有优化空间
- 60%: 性能较差，需要优化
- < 60%: 性能极差，无法使用

**检查项**:
- [ ] API 响应时间 < 500ms
- [ ] 页面加载时间 < 3s（首屏）
- [ ] Lighthouse 性能评分 > 80
- [ ] 无 N+1 查询
- [ ] 无内存泄漏
- [ ] 图片懒加载
- [ ] 代码分割

#### 2.4 安全性 (15%)

**评分标准**:
- 100%: 无安全漏洞，所有安全检查通过
- 90%: 无严重漏洞，少量警告
- 80%: 无严重漏洞，有一些警告
- 70%: 有中等风险漏洞
- 60%: 有高风险漏洞
- < 60%: 有严重安全漏洞

**检查项**:
- [ ] 无 SQL 注入风险（使用 Drizzle sql 模板）
- [ ] 无 XSS 风险
- [ ] 无 CSRF 风险
- [ ] 敏感信息不硬编码
- [ ] 权限验证完整
- [ ] 文件上传安全
- [ ] 速率限制配置

#### 2.5 可维护性 (10%)

**评分标准**:
- 100%: 代码结构清晰，测试覆盖 > 90%
- 90%: 代码结构清晰，测试覆盖 > 80%
- 80%: 代码结构良好，测试覆盖 > 70%
- 70%: 代码结构一般，测试覆盖 > 60%
- 60%: 代码结构较差，测试覆盖 > 50%
- < 60%: 代码结构差，测试覆盖 < 50%

**检查项**:
- [ ] 代码结构清晰
- [ ] 模块职责单一
- [ ] 依赖管理合理
- [ ] 测试覆盖率 > 70%
- [ ] 测试用例充分

### 3. 评分计算

```
总分 = 功能完整性 × 0.30 +
      代码质量 × 0.25 +
      性能表现 × 0.20 +
      安全性 × 0.15 +
      可维护性 × 0.10
```

### 4. 通过标准

#### 4.1 最低通过标准
- **总分 >= 80 分**
- **安全性 >= 90 分**（安全是一票否决）
- **无严重 Bug**（功能完整性 < 70 分视为有严重 Bug）

#### 4.2 优秀标准
- 总分 >= 90 分
- 所有维度 >= 85 分

### 5. 失败处理

#### 5.1 回退重修
- 总分 < 80 分
- 安全性 < 90 分
- 功能完整性 < 70 分
- 有严重 Bug

**流程**: 回退到 N3 重修

#### 5.2 重修次数限制
- **最多重修 3 次**
- 第 3 次重修后仍不通过：**人工介入**

**人工介入流程**:
1. 生成详细的问题报告
2. 列出所有未通过的项目
3. 提供具体的改进建议
4. 等待人工审核和决策

#### 5.3 条件通过
- 总分 >= 80 分
- 但有 1-2 个非关键问题

**处理**: 
- 记录问题到 `ISSUES.md`
- 标记为"条件通过"
- 后续迭代中修复

## 强制规则

- **必须运行已配置的校验**: `pnpm check-types`、`pnpm build`
- **必须检查安全性**: 安全性 < 90 分不通过
- **必须生成评估报告**: 报告必须详细、可追溯
- **必须记录问题**: 所有发现的问题记录到 ISSUES.md
- 接入 Vitest 后启用：运行所有测试 + 覆盖率 < 70% 不通过

## 评估时机

- **N6 节点**: 必须执行完整评估
- **重修后**: 重新执行完整评估
- **发布前**: 执行完整评估
- **紧急修复后**: 执行简化评估（功能 + 安全）

✓ 评估通过后，更新 specs/TASKS_BACKLOG.md「当前执行状态」中的「当前 Node」为 `N7: 上下文管理`。
✓ 评估失败则回退到 N3 重修，并记录失败原因。
